2. INTRODUCCIÓN

2.1 Propósito del Documento

El presente documento tiene como propósito especificar de manera completa y detallada los requisitos de software para el Sistema de Reporte de Problemas de Infraestructura del Instituto Técnico Superior Comunitario (ITSC). Esta especificación de requisitos de software (SRS) sirve como un acuerdo formal entre el equipo de desarrollo y las partes interesadas, describiendo las funcionalidades, restricciones y comportamientos esperados del sistema.

Este documento está dirigido a los evaluadores de la asignatura SOF-111 (Construcción de Software), así como a futuros desarrolladores que puedan dar continuidad al proyecto en cuatrimestres posteriores.

2.2 Descripción General del Sistema

El Sistema de Reporte de Problemas de Infraestructura es una aplicación web diseñada para digitalizar y optimizar el proceso de comunicación entre la comunidad educativa del ITSC y el departamento de mantenimiento. La plataforma permite a estudiantes, profesores y personal administrativo reportar incidencias relacionadas con la infraestructura del instituto de manera rápida y sencilla.

2.3 Tecnologías Utilizadas

PHP 8.2+ Lenguaje de programación del backend
Laravel 12.x Framework PHP para desarrollo web
MySQL 8.0+ Sistema de gestión de base de datos

Blade Laravel Motor de plantillas

  2. Personal de Mantenimiento: Gestiona y actualiza reportes
  3. Estudiante/Profesor: Crea reportes y consulta los propios

3. PROBLEMA A RESOLVER

3.1 Contexto Actual

El proceso de reporte y atención de problemas de infraestructura se ha llevado a cabo tradicionalmente mediante métodos manuales y descentralizados, presentando las siguientes limitaciones:

3.2 Limitaciones Identificadas

# Limitación Descripción
1 Descentralización Reportes en múltiples puntos físicos sin registro centralizado
2 Pérdida de información Formularios en papel se extravían o deterioran
3 Falta de seguimiento El reportante no conoce el estado de su reporte
4 Ineficiencia temporal Tiempo excesivo entre reporte y atención
5 Ausencia de priorización Sin mecanismo para priorizar reportes urgentes
6 Falta de estadísticas Sin reportes estadísticos para decisiones

3.3 Impacto del Problema

En Estudiantes y Profesores:

  • Frustración al reportar problemas
  • Percepción negativa de la institución
  • Condiciones educativas afectadas

En Personal de Mantenimiento:

  • Carga administrativa excesiva
  • Dificultad para priorizar
  • Problemas de coordinación

En Administración:

  • Falta de datos para planificación
  • Imposibilidad de monitoreo
  • Quejas de la comunidad

4. OBJETIVO DEL SISTEMA

4.1 Objetivo General

Desarrollar e implementar un sistema web de gestión de reportes de infraestructura basado en la arquitectura MVC utilizando el framework Laravel, que permita a la comunidad del ITSC reportar, dar seguimiento y gestionar eficientemente las incidencias, centralizando la información en una base de datos única y proporcionando herramientas estadísticas para la toma de decisiones.

9. CASOS DE USO DETALLADOS

9.1 CU-01: Iniciar Sesión

Actor: Todos los usuarios
Precondición: Usuario registrado en BD
Postcondición: Usuario autenticado, sesión activa, redirigido a dashboard

Flujo Principal:

  1. Usuario navega al sistema
  2. Usuario hace clic en "Iniciar Sesión"
  3. Usuario ingresa email
  4. Usuario ingresa contraseña
  5. Usuario hace clic en "Ingresar"
  6. Sistema valida credenciales
  7. Sistema verifica email en BD
  8. Sistema verifica contraseña con bcrypt
  9. Sistema determina el rol del usuario
  10. Sistema redirige al dashboard correspondiente

Flujo Alternativo (Credenciales Incorrectas):

  1. Usuario ingresa credenciales incorrectas
  2. Sistema detecta error
  3. Sistema muestra: "Estas credenciales no coinciden con nuestros registros"
  4. Usuario corrige y reintenta

Excepciones:

  • Usuario no existe → Mensaje genérico de error
  • Cuenta inactiva → "Su cuenta ha sido desactivada"
  • BD no disponible → Error 500
  • Demasiados intentos → Bloqueo temporal (1 min)

9.2 CU-02: Crear Reporte de Incidencia

Actor: Estudiante/Profesor
Precondición: Usuario autenticado
Postcondición: Ticket creado en estado "Pendiente", notificaciones generadas

Flujo Principal:

  1. Usuario navega a reportes
  2. Usuario hace clic en "Crear Nuevo Reporte"
  3. Usuario selecciona ubicación
  4. Usuario ingresa descripción (mín. 10 caracteres)
  5. Usuario selecciona prioridad
  6. Usuario opcionalmente sube fotografía
  7. Usuario hace clic en "Guardar"
  8. Sistema valida datos (FormRequest)
  9. Sistema guarda en tabla tickets
  10. Sistema genera notificaciones
  11. Sistema muestra mensaje de éxito
  12. Sistema redirige al listado

9.3 CU-03: Gestionar Estado del Ticket

Actor: Mantenimiento / Administrador
Precondición: Ticket existente, usuario autenticado con permisos
Postcondición: Estado actualizado, notificación al usuario dueño

Flujo Principal:

  1. Usuario navega a lista de reportes
  2. Usuario identifica ticket a actualizar
  3. Usuario hace clic en el ticket
  4. Usuario hace clic en "Editar"
  5. Usuario selecciona nuevo estado
  6. Usuario hace clic en "Guardar Cambios"
  7. Sistema valida permisos (middleware)
  8. Sistema actualiza registro en BD
  9. Sistema genera notificación al usuario
  10. Sistema muestra mensaje de éxito

13. ARQUITECTURA MVC

13.1 Introducción

El patrón Modelo-Vista-Controlador (MVC) separa la lógica de negocio en tres componentes: Modelo (datos y lógica de negocio), Vista (presentación) y Controlador (flujo de la aplicación). Este patrón facilita el mantenimiento, testing y escalabilidad del código.

13.2 Implementación en Laravel

Componente Ubicación Responsabilidad
Modelo app/Models/ Acceso a datos, lógica de negocio
Vista resources/views/ Presentación (Blade templates)
Controlador app/Http/Controllers/ Flujo de la aplicación
Rutas routes/web.php Mapeo de URLs a controladores

13.3 Flujo del Sistema

  1. Usuario hace petición HTTP
  2. Router dirige al Controlador (routes/web.php)
  3. Controlador solicita datos al Modelo
  4. Modelo consulta Base de Datos (Eloquent ORM)
  5. Modelo retorna datos al Controlador
  6. Controlador pasa datos a la Vista
  7. Vista se renderiza en el navegador (Blade → HTML)
[ESPACIO PARA DIAGRAMA DE FLUJO MVC] Flujo: Usuario → Navegador → Router → Middleware → Controlador → Modelo → BD → Vista → Navegador

15. ANEXOS

15.1 Script SQL

-- Creación de la base de datos
CREATE DATABASE IF NOT EXISTS sistema_reportes 
CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE sistema_reportes;

-- Tabla: roles
CREATE TABLE roles (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL
) ENGINE=InnoDB;

-- Tabla: users
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,
    role_id BIGINT UNSIGNED NOT NULL DEFAULT 3,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,
    FOREIGN KEY (role_id) REFERENCES roles(id)
) ENGINE=InnoDB;

-- Tabla: tickets
CREATE TABLE tickets (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    location VARCHAR(150) NOT NULL,
    description TEXT NOT NULL,
    priority ENUM('baja','media','alta') DEFAULT 'media',
    status ENUM('pendiente','en_proceso','resuelto') DEFAULT 'pendiente',
    photo_path VARCHAR(255) NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- Datos iniciales
INSERT INTO roles (name) VALUES ('admin'), ('mantenimiento'), ('usuario');
